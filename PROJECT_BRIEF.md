# Project Brief — The Discrepancy Desk

## One-Liner

**The Discrepancy Desk** is a fictional/parody public brand plus a local human-approved anomaly archive, research, drafting, and editorial-control system for weird history, conspiracy lore, disputed claims, anomalies, and hidden-timeline content.

The Desk may preserve and publish clearly labeled speculative, folkloric, implausible, disputed, and unresolved material. It is not required to prove or disprove every theory. It must preserve where material came from, distinguish source material from Desk inference where material, and never fabricate the supporting record. See `00-doctrine/editorial-anomaly-archive-direction.md`.

## Public Account Concept

The public account is voiced through **Quinton Clearance**, a fictional former records custodian associated with **Basement 1, Level 7** / **B1-L7**.

Quinton is not a field agent. He is not a screaming conspiracy influencer. He is a tired bureaucratic archive clerk who processes absurd files with deadpan formality.

Core public vibe:

```text
weird history + conspiracy lore + archive clerk + dry paperwork comedy
```

## Private System Concept

The private system helps the user capture topics, sources, posts, replies, article links, videos, and weird leads; store them; draft content; review claims; approve posts; track published output; and learn from metrics.

It should begin local-first and simple.

Preferred early stack:

```text
FastAPI + SQLite + Alembic + Jinja/HTMX
Manual posting/copy-paste at first
Raw capture first
X API read-only probe before schema finalization
Obsidian-compatible research vault later
Qdrant later, after enough notes exist
```

## What This Is Not

This is not:

- an autonomous X bot
- an AI reply bot
- an engagement pod
- a fake insider account
- a government impersonation account
- a generic SEO dashboard
- a scraper farm
- a formal fact-checking service or truth-rating system
- a court of final factual authority
- a system that must resolve every claim before editorial use
- a full strategy engine
- a React-heavy project before need exists
- a Postgres-first architecture before local proof

## Success Definition

Early success means:

1. Public brand exists cleanly.
2. Planning docs are organized.
3. X account is created safely.
4. Source/research workflow is defined.
5. API payloads are inspected before schema decisions.
6. Local dashboard can capture, draft, review, and record posts.
7. Human approval remains mandatory.
